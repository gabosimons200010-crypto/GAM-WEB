# 6. Estrategia de IA (el núcleo del producto)

Muchos comerciantes de Gamarra no tienen conocimientos técnicos; **la plataforma debe hacer casi todo
automáticamente**. La IA convierte "subo fotos" en "tengo productos publicados y vendibles".

> Las referencias de modelos y precios de Claude se basan en el catálogo vigente del Anthropic/Claude API
> (ver §6.7). El pipeline es **multi-proveedor** (Claude Vision y OpenAI Vision) con enrutamiento por costo
> y confianza, de modo que cualquier proveedor puede ser primario o *fallback*.

## 6.1 Arquitectura del pipeline de catalogación

```
                                      ┌──────────────────────────────────────────────┐
  Vendedor sube N fotos               │              worker-ai (cola BullMQ)          │
  (móvil/web, 10..500)  ──┐           │                                              │
                          │  presigned │  1. Pre-proceso (worker-media):              │
              POST /ai/batches ───────▶│     remoción de fondo + WebP + thumbnails    │  IA-004
                          │           │     (modelo especializado, NO el LLM)        │
  S3/R2 (originales) ◀────┘           │  2. Dedupe perceptual (pHash) ───────────────│  IA-005
                                      │  3. Análisis de visión (LLM, structured out) │  IA-001
                                      │     → atributos + género + colores + estilo  │
                                      │  4. Generación de copy comercial             │  IA-002
                                      │     → nombre + descripción + tags + keywords │
                                      │  5. SKU + código interno + barras opcional   │  IA-006
                                      │  6. Embedding (texto+imagen) → OpenSearch     │  búsqueda
                                      │  7. Crea Product DRAFT + AIDraftReady          │
                                      └──────────────────────┬───────────────────────┘
                                                             ▼
                              Vendedor revisa borrador → confirma/edita → ProductPublished
                                            (human-in-the-loop obligatorio)
```

Todo corre **en segundo plano** (`RNF-ESC-003`, `IA-003`). El vendedor ve el progreso (`processed/total`)
y recibe notificación al completarse. Nada bloquea la subida.

## 6.2 IA-001 — Análisis de prendas (extracción estructurada)

Para cada imagen se invoca un modelo de visión con **structured outputs / strict tool use**, garantizando
que la respuesta valide contra un JSON Schema. Esto evita parseo frágil y mantiene consistencia de taxonomía.

Esquema de salida (resumen):
```json
{
  "type": "object",
  "properties": {
    "garmentType":  { "type": "string" },                 // polo, vestido, casaca…
    "categoryId":   { "type": "string" },                 // mapea a taxonomía Category
    "subcategory":  { "type": "string" },
    "gender":       { "enum": ["HOMBRE","MUJER","NINO","NINA","UNISEX"] },
    "material":     { "type": "string" },                 // material probable
    "colors":       { "type": "array", "items": { "type": "string" } },
    "style":        { "type": "string" },
    "season":       { "type": "string" },
    "cut":          { "type": "string" },                 // corte
    "seoTags":      { "type": "array", "items": { "type": "string" } },
    "confidence":   { "type": "number" }                  // 0..1 autoevaluación
  },
  "required": ["garmentType","categoryId","gender","colors","confidence"],
  "additionalProperties": false
}
```

El `categoryId` se restringe a la taxonomía real (se inyecta la lista de categorías válidas en el prompt),
evitando que la IA invente categorías.

## 6.3 IA-002 — Generación automática de contenido comercial

Con los atributos de IA-001 más la imagen, un segundo paso (o el mismo en una sola llamada con dos campos
de salida) genera: **nombre de producto** atractivo, **descripción comercial**, **tags**, **categorías** y
**palabras clave SEO**. El copy se genera en español peruano, orientado a venta, con longitud controlada.

## 6.4 IA-003 — Carga masiva (10/50/100/500 fotos)

- Subida directa a S3/R2 con URLs prefirmadas (no pasa por el `api`).
- Cada imagen es un *job* en la cola `ai`. Concurrencia limitada por *rate limits* del proveedor y
  presupuesto.
- **Cargas masivas no urgentes usan la Batch API** del proveedor (50 % de descuento) cuando el volumen lo
  justifica; cargas pequeñas/interactivas usan la API síncrona para feedback rápido.
- Resultado: **borradores automáticos** que el vendedor revisa. Estado del lote visible en tiempo real.

## 6.5 IA-004 — Remoción de fondo y optimización

**Decisión (ADR-06):** la segmentación de fondo NO la hace el LLM de visión. Se usa un modelo especializado
(Bria RMBG / BiRefNet self-hosted en GPU, o servicio externo como remove.bg) que produce calidad de catálogo.
Se generan **tres variantes** por foto y se guardan en `ProductMedia`:
1. **Original** (respaldo).
2. **Sin fondo** (PNG transparente).
3. **Optimizada para catálogo** → **WebP** multi-resolución (móvil/desktop), **< 200 KB** (`RNF-PERF-003`),
   con `sharp`. Servidas por CDN (`RNF-ESC-002`).

## 6.6 IA-005 / IA-006 / IA-007

- **IA-005 Dedupe:** se calcula un **perceptual hash (pHash)** y un **embedding de imagen**. Antes de crear
  el borrador, se busca similitud contra los productos existentes de la tienda. Si supera un umbral, se
  sugiere **fusionar / actualizar stock / ignorar** (no se decide solo; el vendedor elige).
- **IA-006 SKU automático:** se genera SKU determinístico por producto y por variante
  (p.ej. `{storeCode}-{categoryCode}-{seq}-{size}{color}`), código interno y, opcionalmente, código de barras.
- **IA-007 Importación Excel:** el vendedor descarga plantilla, la completa y la sube (`ImportJob`). El
  sistema valida fila por fila (máx. 200 por carga, `RF-SHOP-004`), muestra errores antes de confirmar y
  permite **relacionar imágenes** con filas para disparar el pipeline de IA donde falten datos.

## 6.7 Selección de modelos, costo y enrutamiento

**Estrategia en cascada por costo/confianza** (ADR-05). Modelos Claude del catálogo vigente del Claude API
(precios por millón de tokens, entrada/salida):

| Rol | Modelo sugerido | Precio (in/out por 1M tok) | Uso |
|-----|-----------------|----------------------------|-----|
| Bulk económico (primer pase masivo) | `claude-haiku-4-5` | $1.00 / $5.00 | Mayoría de imágenes en cargas grandes |
| Calidad por defecto | `claude-sonnet-4-6` | $3.00 / $15.00 | Productos de catálogo estándar |
| Escalado ante baja confianza / atributo crítico | `claude-opus-4-8` | $5.00 / $25.00 | Reintento cuando `confidence` < umbral |

OpenAI Vision se usa como proveedor alterno/fallback con la misma interfaz (`VisionPort`).

**Controles de costo:**
- **Batch API** (−50 %) para `IA-003` masivo no urgente.
- **Prompt caching** del esquema/instrucciones fijas (el JSON Schema y la taxonomía no cambian entre
  imágenes) → grandes ahorros en el componente de entrada repetida.
- **Structured outputs** evitan reintentos por formato inválido.
- **Umbral de confianza:** solo se escala al modelo caro cuando hace falta.
- **Presupuesto y telemetría:** `AIAnalysis.costUsd` registra el costo por análisis; dashboards en Grafana
  alertan si el costo/producto supera el objetivo.

> Nota de implementación: la integración usará el SDK oficial de Anthropic (`@anthropic-ai/sdk`) con
> *adaptive thinking* desactivado/bajo para tareas de extracción (no requieren razonamiento extendido), y
> *structured outputs* (`output_config.format`) o *strict tool use* para garantizar el JSON. El enrutamiento
> de modelos vive detrás de un puerto `VisionPort` para poder cambiar proveedor sin tocar el dominio.

## 6.8 Búsqueda inteligente (semántica y por imagen)

- **Semántica:** se generan embeddings de producto (texto y/o imagen) y se indexan en OpenSearch (k-NN).
  Consultas como *"polo oversize negro"* o *"casaca invierno hombre"* combinan BM25 (léxico) + vector
  (semántico) en una búsqueda híbrida, sin depender de coincidencia exacta.
- **Por imagen:** el usuario sube una foto → se calcula su embedding → k-NN encuentra productos similares.
- **Autocompletado** (`RNF-PERF-002`, < 200 ms): *completion suggester* / edge n-grams sobre nombres,
  marcas y tags, tolerante a errores ortográficos (`RF-CAT-001`).

## 6.9 Human-in-the-loop y calidad

- **Ningún producto se publica solo.** La IA crea **borradores** (`Product.status = DRAFT`); el vendedor
  revisa, corrige y publica. Esto respeta la realidad del comerciante y evita publicar errores.
- Productos de tiendas no verificadas pasan además por la **cola de moderación** del admin (`RF-ADM-002`).
- Métricas de calidad de IA: tasa de aceptación de borradores sin edición, tasa de corrección por atributo,
  costo por producto, latencia del lote. Retroalimentan el ajuste de prompts y umbrales.

## 6.10 Resiliencia del subsistema de IA

- Si un proveedor de visión falla o se satura, el `VisionPort` reintenta con *backoff* y conmuta al
  proveedor alterno.
- Los jobs son idempotentes (clave por `imageHash`): reprocesar no duplica.
- El pipeline degrada con elegancia: si la IA no logra clasificar, crea un borrador "incompleto" para que el
  vendedor lo complete manualmente, en lugar de fallar la carga entera.
